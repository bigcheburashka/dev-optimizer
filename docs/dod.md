# DoD и формат результатов

## Общий Definition of Done для MVP

MVP считается готовым, если одновременно выполнены все условия:

1. Есть рабочий CLI.
2. Есть единый markdown report.
3. Поддержаны 3 домена:
   - CI/CD
   - dependencies
   - Docker
4. Есть единая модель findings.
5. Есть минимум один safe fix path.
6. Есть demo script.
7. Есть test fixtures.
8. Нет критичных ложных обещаний в estimates.

---

## Общий DoD для любой задачи

Любая задача считается завершённой только если:
- есть код или документированный артефакт;
- есть понятный результат, который можно показать;
- есть критерий проверки;
- изменение не ломает соседние части;
- зафиксированы ограничения и known gaps.

---

## DoD по ключевым блокам

### CLI
- команды запускаются;
- help самодостаточный;
- ошибки читаемые;
- есть smoke tests.

### Repo discovery
- корректно работает на неполных репозиториях;
- умеет явно сказать, что найдено, а что нет.

### Findings engine
- все домены возвращают findings в едином формате;
- у finding есть evidence, confidence и suggested action.

### Report
- report можно отправить менеджеру/разработчику;
- есть top findings;
- есть quick wins;
- есть раздел manual fixes.

### Safe fixes
- есть dry-run;
- есть diff preview;
- risky fixes не применяются автоматически.

### Demo
- воспроизводится на 3 demo repos;
- занимает не больше 5 минут.

---

## Обязательный формат результата по каждому этапу

Каждый этап должен заканчиваться сообщением/документом такого вида:

### 1. Что сделано
Короткий список реализованных вещей.

### 2. Что работает
Что уже реально можно запустить/проверить.

### 3. Что не сделано
Явный список незавершённого.

### 4. Риски / ограничения
Какие места хрупкие или требуют ручной проверки.

### 5. Следующий шаг
Один ближайший логичный шаг, без размывания.

## Минимальный пакет артефактов к концу MVP

- `README.md`
- `docs/mvp-scope.md`
- `docs/architecture.md`
- `docs/finding-schema.md`
- `docs/demo-script.md`
- `examples/` или `fixtures/`
- CLI команда `analyze`
- CLI команда `fix --safe`
- markdown/json output examples

---

## Что считается провалом этапа

Этап нельзя считать завершённым, если:
- есть только код без проверяемого вывода;
- есть только идеи без артефактов;
- следующий исполнитель не понимает, что запускать;
- нет явного списка ограничений;
- unsafe logic замаскирована как safe.
